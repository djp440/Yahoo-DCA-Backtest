# Claude 项目开发规范

## ⚠️ 强制遵守规则
所有开发工作必须严格遵守以下文档要求，不得擅自变更需求和设计：

### 1. 需求规范 - 严格遵守 `/doc/PRD.md`
- 功能实现完全按照PRD定义的需求范围
- MVP阶段仅实现P0级功能，P1/P2功能后续迭代
- 模块划分、数据流、技术架构必须与PRD一致
- 日志模块为P0级基础能力，必须第一阶段实现

### 2. 设计规范 - 严格遵守 `/doc/UI_DESIGN.md`
- UI风格必须为"明亮清新苹果设计风格"
- 色彩、字体、间距、圆角、阴影完全使用设计系统定义的Token
- 动画效果必须符合要求，提供弹性动画、视差效果的流畅体验
- 布局严格采用"顶部菜单栏 + 下方工作区"结构
- 图表组件使用Lightweight Charts，配色完全遵循设计规范

### 3. 项目状态维护
- 每次session开始必须先读取 `/doc/STATUS.md` 中的「任务目标」，防止需求漂移
- 每完成一个里程碑/关键步骤，必须更新 `/doc/STATUS.md` 的「已完成步骤」和「当前状态」
- 开发中遇到的问题、踩过的坑必须记录到「已知的坑」章节，避免后续重复踩坑

---

## 🛠️ 技术栈强制要求
| 技术 | 版本要求 | 说明 |
|------|----------|------|
| 桌面框架 | Tauri 2.0 | 打包为Windows exe，使用Rust作为后端 |
| 前端框架 | Vue 3 + Vite | 使用组合式API，TypeScript语言 |
| UI样式 | Tailwind CSS 3.x | 完全按照UI设计文档配置主题 |
| 图表库 | Lightweight Charts | 专业金融图表渲染 |
| 数据源 | yahoo-finance2 | Yahoo Finance数据接口 |
| 本地存储 | SQLite | 数据缓存、日志存储、回测记录存储 |
| 状态管理 | Pinia | 全局状态管理 |

---

## 📋 开发流程要求
1. **需求确认**：每次开发前确认需求范围，不做PRD以外的功能
2. **实现前参考**：开发UI前必须先查阅UI设计文档的规范
3. **日志埋点**：所有核心功能必须添加日志埋点，使用统一的日志SDK
4. **状态更新**：完成关键节点后必须更新STATUS.md文档
5. **编码规范**：遵循TypeScript/Rust最佳实践，代码清晰易读

---

## ❌ 禁止事项
- 禁止擅自修改需求、添加PRD以外的功能
- 禁止违反UI设计规范，使用不符合要求的配色、布局、交互
- 禁止在开发阶段引入不必要的第三方依赖
- 禁止修改STATUS.md中的「任务目标」和历史「已完成步骤」记录

---

## 🛣️ 开发执行路径（强制遵守）
### 1. 任务执行顺序
严格按照`TODO.md`中的任务顺序从上到下执行，**禁止跳步**，前一个任务验收通过后才能开始下一个任务。

### 2. 单任务执行规则
- 每个任务只允许修改`对应文件`列指定的单个文件，不得修改其他无关文件
- 开发完成后必须严格对照`验收标准`进行验证，完全符合后才能标记为完成
- 任务开发过程中遇到问题，记录到`doc/STATUS.md`的「已知的坑」章节

### 3. 状态更新规范
- 单个任务完成并验收通过后，更新`TODO.md`中对应任务的状态为✅ 已完成
- 每个里程碑（阶段）全部完成后，更新`doc/STATUS.md`的「已完成步骤」和「当前状态」
- 每次会话结束前，确保所有已完成任务的状态都已更新到文档中

### 4. 问题处理流程
- 遇到技术问题首先查阅「已知的坑」章节，确认是否有历史解决方案
- 未解决的问题需要详细记录到「已知的坑」，包含问题描述、复现步骤、尝试过的方案
- 阻塞性问题需要及时反馈，不要擅自修改需求绕过问题
